# Picking up a flat washer from a soft surface — with Nebius Serverless AI Jobs

**Nebius Serverless AI Builders Challenge — Physical AI & Robotics** · `#NebiusServerlessChallenge`

A flat washer lying face-down is one of the hardest picks for a parallel-jaw gripper: there is
nothing to grab. This project teaches a UR10e in Isaac Sim to **flip the washer off a compliant
surface** — press one edge down, the other tips up, pinch, lift — and uses **Nebius Serverless AI
Jobs** to run everything that is too heavy for one workstation: dataset generation, evaluation,
**material × maneuver co-optimization** (38 H100 jobs), and an overnight
**evolution-strategies training** whose model checkpoints ride home in the job logs.

📺 **Video walkthrough (9 min):** https://youtu.be/IAawtx7Snow
📝 **Engineering log:** [`data/reports/`](data/reports/) — day-by-day reports, honest negatives included.

## Results

| Milestone | Result |
|---|---|
| Grasp-success scorer (numpy MLP, 26 state features) | val **AUC 0.823** on a 4,000-attempt self-generated dataset |
| Env-tuned pick under jitter (±1.5 mm, ±20° roll) | **17/18 (94 %)**, zero flips — tuned *in the deploy env* by chained Nebius jobs (≈ $1.80) |
| Material × maneuver co-optimization (9 sourceable materials) | winner: **soft PU foam, ~60 kPa / 10 mm** → **89.4 % ± 4.1**, 1.4 % flips under sourcing noise (+20 pp over incumbent) |
| Closed-loop residual policy (1,603-param MLP, OpenAI-ES) | honest negative: parity with open-loop θ\* (76.9 % vs 75.9 % wide-DR); θ\* stays deployed |

## How Nebius is used

- **Serverless AI Jobs (H100)** run the containerized, headless Isaac Sim workloads:
  data generation (`jobs/gen_data.py`), evaluation (`jobs/eval_sort.py`), env-in-the-loop CEM
  tuning (`tools/arena_eval.py` via `job/chain_env_tune.sh`), the material sweep
  (`job/stage1_materials.sh` … `stage3_verdicts.sh`), and ES training
  (`tools/resid_policy.py` via `job/auto_relaunch_es.sh`).
- **Jobs are sized ~14 min** to fit under the platform's job window; each job seeds the next
  (chained resumption). During ES training the **weights are dumped base64 into the job logs
  every update** — the log line *is* the model artifact, so a killed job loses ≤ 1 update.
- **Scale-to-zero economics:** test locally first, then pay by the hour only while a job runs —
  and run several chains in parallel on one home GPU budget.
- A CPU **Serverless Endpoint** (`graspsvc`, `docker/`) serves the trained scorer over HTTPS.

## Reproduce it

Prereqs: Docker + [Nebius CLI](https://nebius.com) (authenticated), an Isaac Sim 6.0 install for
local runs (`ISAAC=~/TOOLS/isaac-sim/python.sh` or your path), and your own Nebius ids:

```bash
export NEBIUS_REGISTRY=<your container registry>      # cr.<region>.nebius.cloud/<id>
export NEBIUS_PROJECT_ID=<your project id>            # project-…
# object storage credentials for job outputs:
export AWS_ACCESS_KEY_ID=… AWS_SECRET_ACCESS_KEY=…
```

```bash
# 1. local smoke (no cloud, no cost): boot the slim env and run one grasp
"$ISAAC" tools/smoke_bt.py

# 2. build + push the job image (Taskfile.yml holds the exact commands)
task build:push

# 3. small validation job → confirm records land in your bucket & the job scales to zero
task job:gen N=50 BATCH=50

# 4. the real thing
task job:gen N=5000 BATCH=200          # labelled grasp dataset
task job:eval POLICY=heuristic         # "before" sort-success report
bash job/stage1_materials.sh           # material × maneuver sweep (H100 jobs)
bash job/chain_env_tune.sh             # env-in-the-loop pick tuning chain
bash job/auto_relaunch_es.sh           # ES training with log-line checkpoints
```

**Expected outputs:** `records.jsonl` dataset in your bucket → `model.npz` scorer (AUC ≈ 0.82) →
per-material tuning verdicts (PU-soft ≈ 89 % winner) → ES weight checkpoints recoverable from job
logs (`WSAVE` lines). Every number above is regenerated by these scripts; seeds and panel sizes are
in the report pages.
